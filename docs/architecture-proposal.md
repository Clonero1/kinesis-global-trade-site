# Архитектурное предложение Kinesis Global Trade

## 1. Цель архитектуры

Сайт должен стартовать как B2B/B2C-каталог запчастей с заявками, быстрым поиском по артикулу, массовым поиском и импортом товаров. При этом модель данных и границы модулей должны сразу учитывать будущий маркетплейс: продавцов, раздельные остатки, модерацию карточек, комиссии и кабинеты продавцов.

Ключевое правило: артикул и каталожные номера являются главным основанием для поиска, сопоставления, обновления и контроля дублей. Название товара не должно использоваться как надежный идентификатор и не должно автоматически делать товары аналогами.

## 2. Рекомендуемый стек

Рекомендуемый вариант: Next.js + PostgreSQL + Prisma + Redis + S3-совместимое хранилище.

| Уровень | Рекомендация | Зачем |
|---|---|---|
| Frontend и backend | Next.js App Router, TypeScript | Один кодовый контур для публичного сайта, личного кабинета, админки и API. Хорошо подходит для SEO и будущего масштабирования. |
| База данных | PostgreSQL | Надежные связи, транзакции, индексы по артикулам, заказы, клиенты, импорт, аудит. |
| ORM | Prisma | Типизированная схема БД, миграции, понятная поддержка. |
| Поиск | PostgreSQL trigram/full-text на первом этапе, OpenSearch/Meilisearch позже | На старте достаточно точного и нормализованного поиска по артикулам; отдельный поисковый движок можно подключить при росте каталога. |
| Очереди | Redis + BullMQ или аналог | Импорт файлов, синхронизация с 1С, генерация XML для Авито, пересчет индексов и логирование интеграций. |
| Файлы | S3-совместимое хранилище | Фото товаров, импортированные файлы, сгенерированные документы и XML-выгрузки. |
| Авторизация | Email/пароль + роли, позже SSO при необходимости | Покупатели, юридические лица, менеджеры, администраторы, продавцы маркетплейса. |
| Интеграции | Отдельный integration-layer внутри приложения | Изоляция 1С, Битрикс24, Авито, СДЭК, Деловых Линий от основной бизнес-логики. |

Альтернатива: Laravel + PostgreSQL подойдет, если команда сильнее в PHP и нужна классическая админка. 1С-Битрикс можно рассматривать только если важнее готовая CMS, чем гибкая архитектура маркетплейса; для задач поиска, импорта, качества данных и будущих продавцов он несет риск доработок вокруг платформенных ограничений.

## 5. Логика каталога

Каталог должен хранить одну базовую карточку товара и одно или несколько предложений. Даже если на старте продавец один, предложение должно быть отдельной сущностью, потому что цена, остаток, склад и состояние могут отличаться.

Карточка товара показывает:

- название;
- основной артикул;
- дополнительные каталожные номера;
- производителя;
- применимость, если известна;
- состояние;
- цену;
- остаток;
- склад;
- фото;
- описание;
- кнопку добавления в заявку.

Товары с нулевым остатком не следует автоматически скрывать. Для них нужен статус: нет в наличии, доступно под заказ или скрыто. Решение о скрытии должно быть настройкой.

## 6. Поиск по артикулу

Поиск должен идти в два шага:

1. Нормализация запроса: trim, верхний регистр, удаление повторных пробелов, сохранение исходной строки, расчет вариантов с дефисами и без дефисов.
2. Сопоставление с `product_identifiers.value_normalized`.

Режимы совпадения:

| Режим | Что значит | Действие |
|---|---|---|
| exact | Полное совпадение нормализованного артикула | Можно показывать как найденный товар |
| alias | Совпадение по дополнительному каталожному номеру | Показывать с пометкой найденного номера |
| format_variant | Отличие только в дефисах/пробелах/регистре | Показывать, но хранить исходное написание |
| ambiguous | Несколько товаров с тем же номером | Отправить в ручную проверку или показать выбор |
| no_match | Совпадений нет | Показать статус не найдено и предложить заявку |

Автоматически объединять товары по похожему названию нельзя. Название может использоваться только как дополнительный сигнал для ручной проверки.

## 7. Массовый поиск

Массовый поиск должен поддерживать:

- вставку списка артикулов в текстовое поле;
- загрузку Excel/CSV;
- сохранение исходного артикула пользователя;
- результат таблицей;
- добавление найденных позиций в заявку;
- экспорт результата в Excel на следующем этапе.

Результат по каждой строке:

| Поле | Описание |
|---|---|
| input_article | Исходный артикул пользователя |
| normalized_article | Нормализованное значение |
| matched_article | Найденный артикул или каталожный номер |
| product_name | Название товара |
| price | Цена |
| stock_qty | Остаток |
| warehouse | Склад |
| condition | Состояние |
| status | найдено в наличии / найдено, нет остатка / доступно под заказ / не найдено / требуется проверка |

На первом этапе разумный лимит: до 500 строк через текстовое поле и до 5 000 строк через файл. Большие списки лучше выполнять асинхронно через очередь.

## 8. Импорт Excel, CSV, XML и 1С

Импорт должен быть отдельным пайплайном:

1. Загрузка файла или получение данных из 1С/API.
2. Сохранение исходного файла и создание `import_batch`.
3. Парсинг строк в `import_rows`.
4. Валидация обязательных полей.
5. Нормализация артикулов.
6. Поиск существующего товара по `internal_id`, `article`, дополнительным каталожным номерам.
7. Создание/обновление товара и offer.
8. Фиксация ошибок, дублей и строк для ручной проверки.
9. Отчет по импорту.

Минимальные поля импорта:

| Поле | Обязательность | Комментарий |
|---|---:|---|
| internal_id | желательно | Внутренний код из 1С |
| article | обязательно | Основной артикул |
| additional_articles | желательно | Дополнительные каталожные номера |
| name | обязательно | Название |
| brand | желательно | Производитель |
| category | желательно | Категория |
| condition | обязательно | new, used, remanufactured, repair_after, on_order |
| warehouse | желательно | Склад |
| stock_qty | обязательно | Остаток |
| price | желательно | Цена |
| currency | желательно | Валюта |
| photo_url | желательно | Фото |
| weight, length, width, height | желательно | Нужно для доставки |
| is_active | обязательно | Показывать на сайте |

Ошибки должны быть понятными: не указан артикул, некорректный остаток, найден дубль, неизвестное состояние, не определена категория.

## 9. Личный кабинет покупателя

Личный кабинет вводится после запуска базового каталога. В нем нужны:

- регистрация и авторизация;
- тип клиента: физическое лицо или юридическое лицо;
- хранение реквизитов юрлица;
- история заявок и заказов;
- повтор заказа;
- сохраненные списки артикулов;
- документы по заказам;
- статусы заказов.

Для юридических лиц реквизиты должны храниться отдельно от профиля пользователя, потому что один пользователь может работать с организацией, а в будущем у организации может быть несколько сотрудников.

## 10. Заявки и заказы

На первом этапе основная сущность - заявка. Клиент добавляет позиции из карточки или массового поиска, указывает контакты и комментарий, после чего заявка попадает менеджеру.

Поток:

1. Клиент ищет один артикул или список.
2. Добавляет найденные позиции в список заявки.
3. Указывает количество, контакты, комментарий.
4. Система сохраняет заявку и позиции со снимком цены/остатка.
5. Система отправляет уведомление менеджеру и, при наличии настройки, создает лид в Битрикс24.
6. После обработки менеджер переводит заявку в заказ.

На этапе интеграции с 1С заказ может передаваться в 1С, а счет формироваться на стороне 1С или сайта - это нужно согласовать.

## 11. Интеграция с Битрикс24

Битрикс24 рекомендуется использовать как CRM для обработки заявок, не как источник истины по товарам.

Передавать в лид:

- контактные данные клиента;
- список товаров;
- исходные артикулы из поиска;
- количество;
- комментарий;
- источник обращения: site;
- ссылку на заявку в админке;
- статус массового поиска, если часть позиций не найдена.

Если Битрикс24 недоступен, заявка должна сохраниться на сайте, а интеграция должна повториться через очередь.

## 12. Интеграция с Авито

Авито на первом этапе лучше поддержать через XML-выгрузку.

Поток:

1. Администратор выбирает товары для публикации или правило отбора: активные, в наличии, с фото, с ценой.
2. Система формирует XML.
3. XML сохраняется и доступен по защищенной или публичной ссылке.
4. Логируется количество выгруженных и исключенных товаров.
5. Товары без остатка могут автоматически исключаться из выгрузки, если включено правило.

Дополнительно нужна сверка: товар есть на сайте, но отсутствует в XML; товар есть в XML, но отсутствует в остатках.

## 13. Расчет доставки

Для СДЭК:

- расчет стоимости;
- расчет срока;
- выбор ПВЗ;
- адресная доставка;
- сохранение выбранного тарифа в заказ.

Для Деловых Линий:

- расчет стоимости для тяжелых и крупногабаритных товаров;
- учет веса и габаритов;
- выбор терминала;
- сохранение расчета в заказ.

Если у товара нет веса и габаритов, система должна показывать ручной расчет менеджером, а не давать ложную цену.

## 14. Развитие до маркетплейса

Чтобы не переделывать проект полностью, уже на первом этапе нужно:

- хранить товары отдельно от предложений;
- иметь поле `seller_id` в offer, даже если продавец один;
- хранить источник данных;
- вести историю изменений;
- отделить модерацию карточек от обычного редактирования;
- проектировать роли: покупатель, менеджер, администратор, продавец;
- не смешивать остатки разных складов и продавцов в одной строке товара.

Маркетплейс добавит личный кабинет продавца, загрузку товаров продавцом, модерацию, разделение заказов по продавцам, комиссии и отчеты.

## 15. Потоки данных

| Поток | Источник | Назначение | Комментарий |
|---|---|---|---|
| Товары, цены, остатки | 1С / Excel / CSV / XML | Каталог и offers | 1С желательно сделать главным источником, но Excel оставить резервным каналом. |
| Фото | 1С / админка / URL | S3-хранилище и карточки | Нужно согласовать место хранения оригиналов. |

| Доставка | Сайт + API перевозчиков | Заказ | При отсутствии габаритов - ручной расчет. |
| Документы | Сайт или 1С | Личный кабинет | Источник генерации счета нужно согласовать. |

## 16. Основные риски

| Риск | Вероятность | Влияние | Снижение риска |
|---|---:|---:|---|
| Нет единого источника истины по товарам | высокая | высокое | Согласовать: 1С, сайт или отдельная PIM; зафиксировать владельца данных. |
| Дубли из-за разных написаний артикулов | высокая | высокое | Нормализация, `product_identifiers`, отчеты дублей, ручная проверка. |
| 1С не готова к прямому обмену | средняя | высокое | Начать с Excel/CSV, заложить API и логи обмена. |
| Неполные веса и габариты | высокая | среднее | Для доставки давать ручной расчет, пока данные не заполнены. |
| Автоматическое объединение аналогов по названию | средняя | высокое | Запретить автосклейку по названию, использовать только ручную модерацию. |
| Рост каталога замедлит поиск | средняя | среднее | Индексы PostgreSQL, затем отдельный поисковый движок. |
| Маркетплейс потребует переделки | средняя | высокое | Сразу ввести `seller`, `offer`, роли и модерацию как расширяемые сущности. |
| Интеграции падают и теряют заявки | средняя | высокое | Сначала сохранять заявку в БД, затем отправлять через очередь с повтором. |

## 18. Рекомендуемый первый релиз

Первый релиз должен включать:

1. Публичный сайт с каталогом, категориями, карточками и контактами.
2. Точный поиск по артикулу.
3. Массовый поиск текстом и файлом.
4. Заявку/корзину с отправкой менеджеру.
5. Базовую админку товаров, заявок и импортов.
6. Импорт Excel/CSV с отчетом ошибок.
7. Модель данных с `products`, `identifiers`, `offers`, `warehouses`, `requests`.
8. Подготовленную интеграционную архитектуру для 1С, Битрикс24, Авито и доставки.

Все остальные функции стоит подключать итерациями, не ломая базовую модель каталога и предложений.
