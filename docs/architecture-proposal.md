# Архитектурное предложение

## 1. Зафиксированные решения

Проект разрабатывается на следующем стеке:

- Next.js;
- TypeScript;
- PostgreSQL;
- Prisma;
- Redis/BullMQ для фоновых задач, очередей и повторной отправки интеграционных событий;
- S3-совместимое хранилище для файлов, изображений, документов и XML-выгрузок.

Главным источником товаров, цен, остатков, складов и номенклатуры является 1С.

Главным источником заказов, счетов, договоров и других официальных документов также является 1С.

Сайт не должен быть главным источником складской, бухгалтерской или юридической истины. Он хранит синхронизированную рабочую копию данных, необходимую для каталога, поиска, заявок, личного кабинета и интеграций.

Основная роль сайта:

- публичная витрина;
- каталог товаров;
- точный поиск по артикулу;
- массовый поиск товаров списком;
- прием заявок от покупателей;
- личный кабинет покупателя;
- интеграционный слой между клиентом, 1С, Битрикс24, Авито и службами доставки.

## 2. Распределение ответственности

| Область | Главный источник | Роль сайта |
| --- | --- | --- |
| Номенклатура | 1С | Получает и отображает товары |
| Артикулы | 1С | Использует как главный идентификатор поиска |
| Цены | 1С | Показывает актуальную синхронизированную цену |
| Остатки | 1С | Показывает остатки по складам |
| Склады | 1С | Отображает склад в карточке и результатах поиска |
| Заявки клиентов | Сайт и 1С | Сайт принимает и сохраняет заявку, затем передает в 1С |
| Заказы | 1С | Получает статусы и отображает их в кабинете |
| Счета, договоры, УПД и другие документы | 1С | Получает ссылки или файлы документов |
| CRM-события | Битрикс24 | Передает лиды, сделки или уведомления по согласованному сценарию |
| Доставка | СДЭК, Деловые Линии, 1С | Получает расчет, статусы и передает выбранный способ доставки |
| XML-выгрузка | Сайт на основе данных из 1С | Формирует техническую выгрузку для Авито |

## 3. Общая архитектура

Сайт строится как Next.js-приложение с серверной частью внутри того же проекта. База PostgreSQL используется для хранения локальной копии каталога, заявок, пользователей, логов обмена, очередей, статусов и служебных сущностей.

Prisma используется как основной слой доступа к базе данных.

Интеграции с 1С, Битрикс24, Авито и службами доставки должны быть вынесены в отдельные модули, чтобы бизнес-логика сайта не зависела напрямую от конкретного API внешней системы.

Предлагаемые крупные модули:

| Модуль | Назначение |
| --- | --- |
| Public site | Главная страница, каталог, карточка товара |
| Catalog | Хранение и отображение товаров, категорий, брендов, складов, цен и остатков |
| Article search | Точный поиск по артикулу и нормализованным идентификаторам |
| Bulk search | Массовый поиск товаров по списку артикулов |
| Import | Загрузка тестовых Excel/CSV для разработки и проверки каталога |
| Cart/request | Корзина, заявка, сохранение заявки в базе сайта |
| Admin | Управление товарами тестового контура, импортами, заявками и логами |
| 1C integration | Получение каталога, цен, остатков, статусов и документов; отправка заявок |
| Bitrix24 integration | Передача лидов, сделок или уведомлений |
| Buyer cabinet | История заявок, статусы, документы, данные покупателя |
| Delivery integration | Расчет и статусы доставки СДЭК и Деловые Линии |
| Avito export | XML-выгрузка товаров для Авито |
| Marketplace preparation | Архитектурный задел для будущих продавцов и комиссий |

## 4. Данные каталога

Каталог сайта является синхронизированной копией данных из 1С. Пользовательские сценарии могут читать эти данные из PostgreSQL, но изменение основной номенклатуры, цен, остатков и складов должно происходить в 1С.

Основные сущности каталога:

- товар;
- артикул;
- дополнительные идентификаторы товара;
- бренд;
- категория;
- состояние товара;
- склад;
- остаток по складу;
- цена;
- изображения;
- применяемость;
- дата последней синхронизации с 1С.

Артикул является главным идентификатором товара. Поиск должен в первую очередь опираться на точное совпадение артикула или нормализованного артикула.

Товары нельзя автоматически считать аналогами только из-за похожего названия. Аналоги допускаются только при наличии явной связи из 1С, ручной связи администратора или отдельного подтвержденного справочника аналогов.

Состояние товара хранится отдельно от названия и описания. Минимальные значения:

- новый;
- б/у;
- восстановленный;
- под заказ.

Склад должен отображаться:

- в карточке товара;
- в результатах поиска;
- в массовом поиске;
- в административном просмотре остатков.

## 5. Заявки и заказы

Сайт принимает заявку от клиента и всегда сначала сохраняет ее в PostgreSQL. Это обязательное правило, чтобы заявка не терялась при недоступности 1С.

После сохранения сайт создает интеграционную задачу на отправку заявки в 1С. Отправка выполняется через интеграционный слой и очередь повторных попыток.

Базовый сценарий:

1. Клиент добавляет товары в корзину или заявку.
2. Сайт сохраняет заявку, состав товаров, контактные данные, юридическое лицо и доставку.
3. Сайт фиксирует снимок цен и остатков на момент заявки.
4. Сайт ставит задачу отправки заявки в 1С.
5. 1С создает заказ, счет, договор или другой официальный документ по своим правилам.
6. Сайт получает от 1С внешний номер, статус, ссылки или файлы документов.
7. Клиент видит статус и документы в личном кабинете.

Сайт не должен самостоятельно формировать официальные счета, договоры или бухгалтерские документы. Допускается только отображение документов, полученных из 1С, или ссылок на них.

## 6. Интеграция с 1С

1С является центральной системой для:

- номенклатуры;
- артикулов;
- цен;
- остатков;
- складов;
- заказов;
- счетов;
- договоров;
- официальных документов;
- статусов заказов.

Интеграционный слой сайта должен поддерживать два направления обмена:

| Направление | Назначение |
| --- | --- |
| 1С -> сайт | Товары, цены, остатки, склады, статусы заказов, документы |
| Сайт -> 1С | Заявки, состав заявки, клиент, юридическое лицо, доставка, комментарии |

Недоступность 1С не должна останавливать сайт. Каталог работает на последней успешно полученной копии данных. Заявки сохраняются локально и повторно отправляются через очередь.

Подробное техническое задание на обмен с 1С описано в `docs/onec-integration-spec.md`.

## 7. Поиск по артикулу

Поиск по артикулу должен быть точным и предсказуемым. Для этого нужно хранить:

- оригинальный артикул из 1С;
- нормализованный артикул;
- дополнительные идентификаторы, если они передаются из 1С.

Нормализация артикула может включать:

- приведение к единому регистру;
- удаление пробелов;
- удаление дефисов и разделителей;
- сохранение исходного значения для отображения.

Поиск не должен подменять точный результат похожими товарами без явной пометки. Если точного совпадения нет, сайт может показать близкие совпадения как отдельный блок, но не должен утверждать, что это аналоги.

## 8. Массовый поиск

Массовый поиск принимает список артикулов и возвращает результат по каждой строке входного списка.

Для каждой позиции нужно показать:

- исходный артикул пользователя;
- найденный товар или статус "не найдено";
- название;
- состояние;
- цену;
- остаток;
- склад;
- возможность добавить в заявку.

Если найдено несколько товаров с одним артикулом, но разным состоянием или складом, сайт должен показать варианты явно.

## 9. Административная часть

Базовая админка нужна уже в первом рабочем релизе.

Минимальные разделы:

- товары;
- импорты тестовых данных;
- заявки;
- статусы отправки в 1С;
- логи интеграций;
- ручной повтор отправки заявки;
- просмотр последних синхронизаций.

Админка не должна становиться заменой 1С для ведения номенклатуры, цен и остатков. На ранних этапах допускается тестовый импорт Excel/CSV, но после подключения 1С главным источником данных остается 1С.

## 10. Битрикс24

Битрикс24 используется как CRM-контур, а не как источник складской или товарной истины.

Возможные сценарии:

- создание лида по заявке;
- создание сделки после успешной отправки заявки в 1С;
- передача контактных данных клиента;
- передача комментария менеджеру;
- уведомления об ошибках или важных событиях.

Финальный сценарий нужно согласовать отдельно, чтобы не дублировать ответственность 1С и CRM.

## 11. Доставка

Интеграции со СДЭК и Деловыми Линиями должны использоваться для расчета и отображения вариантов доставки.

Сайт может хранить выбранный способ доставки и передавать его в 1С вместе с заявкой. Если оформление доставки выполняется в 1С, сайт должен получить от 1С статус и трек-номер.

## 12. Авито

XML-выгрузка для Авито формируется сайтом на основе данных, полученных из 1С.

Выгрузка не должна создавать отдельную независимую товарную базу. Если товар, цена или остаток изменились в 1С, эти изменения должны попасть в локальную копию сайта, а затем в XML.

## 13. Подготовка к маркетплейсу

Полноценный маркетплейс не входит в первый рабочий релиз, но архитектура не должна закрывать путь к нему.

На будущее нужно предусмотреть:

- отдельную сущность продавца;
- принадлежность товара продавцу;
- правила модерации;
- комиссии;
- кабинет продавца;
- расширенные статусы заказов;
- разные источники товаров.

При этом на текущем этапе главным источником товаров остается 1С, а не продавцы сайта.

## 14. Ключевые риски

| Риск | Как учитывать |
| --- | --- |
| Неясный формат обмена с 1С | До разработки согласовать протокол, поля, частоту и ответственных |
| Низкое качество артикулов | Ввести нормализацию, хранить оригинал и дополнительные идентификаторы |
| Устаревшие остатки | Синхронизировать остатки чаще товаров и показывать время обновления в админке |
| Недоступность 1С | Сохранять заявки локально и отправлять через очередь |
| Дубли заказов в 1С | Использовать идемпотентный идентификатор заявки сайта |
| Смешение ролей сайта и 1С | Не формировать официальные документы на сайте |
| Автоматические ложные аналоги | Не объединять товары по похожему названию |

## 15. Итоговое решение

Архитектура проекта должна строиться вокруг 1С как главного источника товарной, складской и документальной истины.

Сайт отвечает за быстрый пользовательский интерфейс, каталог, поиск, массовый подбор, прием заявок, кабинет покупателя и интеграционный слой.

Все критичные действия должны проектироваться так, чтобы:

- заявка клиента не терялась;
- данные каталога были синхронизированы с 1С;
- документы формировались в 1С;
- ошибки обмена были видны администраторам;
- будущие интеграции и маркетплейс могли быть добавлены без переписывания базовой архитектуры.
