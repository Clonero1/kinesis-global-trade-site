# Схема базы данных

## 1. Назначение базы

База PostgreSQL хранит рабочую копию данных сайта Kinesis Global Trade для первого рабочего релиза.

Главный источник товаров, цен, остатков, складов и документов - 1С. Сайт хранит локальную синхронизированную копию этих данных, чтобы быстро показывать каталог, карточку товара, точный поиск по артикулу и массовый поиск.

Заявки создаются на сайте. Сайт всегда сначала сохраняет заявку в своей базе, а затем передает ее в 1С через интеграционный слой. Если 1С недоступна, заявка остается в базе сайта и ожидает повторной отправки.

## 2. Каталог

### Product

Главная таблица локальной карточки товара.

Ключевые поля:

- `id` - внутренний идентификатор сайта;
- `onecId` - идентификатор товара в 1С, если он передается напрямую;
- `name` - название товара;
- `description` - описание;
- `applicability` - применяемость;
- `brandId` - связь с брендом;
- `categoryId` - связь с категорией;
- `conditionId` - связь с состоянием товара;
- `sellerId` - задел для будущего маркетплейса;
- `isActive` - можно ли показывать товар;
- `source` - источник последнего создания или обновления;
- `syncedAt` - дата последней синхронизации с источником.

Данные из 1С:

- название;
- описание;
- применяемость;
- бренд;
- категория;
- состояние;
- габариты;
- вес;
- ставка НДС;
- активность товара.

На сайте можно создавать или обновлять товар только в тестовом контуре импорта. После подключения 1С основные поля товара нельзя редактировать на сайте как источник истины.

### ProductIdentifier

Таблица артикулов и других идентификаторов товара.

Ключевые поля:

- `productId` - товар;
- `type` - тип идентификатора: артикул, OEM, кросс-номер, код поставщика, штрихкод;
- `originalValue` - исходное значение;
- `normalizedValue` - нормализованное значение для поиска;
- `isPrimary` - основной артикул;
- `source` - источник.

Назначение:

- один товар может иметь несколько артикулов и дополнительных кодов;
- основной артикул отмечается явно;
- быстрый поиск идет по `normalizedValue`;
- оригинальное значение хранится для отображения пользователю.

Критичные индексы:

- индекс по `normalizedValue`;
- индекс по `type, normalizedValue`;
- уникальность `productId, type, normalizedValue`.

### Brand

Справочник брендов и производителей.

Ключевые поля:

- `name`;
- `slug`;
- `onecId`.

Данные обычно приходят из 1С. На сайте их можно использовать для фильтров и отображения.

### Category

Справочник категорий.

Ключевые поля:

- `name`;
- `slug`;
- `onecId`;
- `parentId`.

Категории поддерживают дерево через связь родитель-дочерняя категория.

### ProductCondition

Справочник состояний товара.

Ключевые поля:

- `code`;
- `name`;
- `description`;
- `sortOrder`.

Минимальные значения enum:

- `NEW` - новый;
- `USED` - б/у;
- `REMANUFACTURED` - восстановленный;
- `ON_ORDER` - под заказ.

Состояние хранится отдельно от названия товара. Название не должно использоваться для определения состояния.

### ProductImage

Изображения товара.

Ключевые поля:

- `productId`;
- `url`;
- `alt`;
- `sortOrder`;
- `isPrimary`;
- `source`.

Изображения могут приходить из 1С или отдельного файлового хранилища.

### ProductAttribute

Характеристики товара.

Ключевые поля:

- `productId`;
- `name`;
- `value`;
- `unit`;
- `sortOrder`;
- `source`.

Используется для технических характеристик, которые не стоит превращать в отдельные колонки `Product`.

### Warehouse

Справочник складов.

Ключевые поля:

- `onecId`;
- `name`;
- `code`;
- `address`;
- `city`;
- `isActive`;
- `syncedAt`.

Склады приходят из 1С. Сайт показывает склад в карточке товара, поиске и массовом поиске.

### ProductStock

Остаток товара по складу.

Ключевые поля:

- `productId`;
- `warehouseId`;
- `quantity`;
- `reservedQuantity`;
- `availableQuantity`;
- `syncBatchId`;
- `syncedAt`.

Остаток вынесен в отдельную таблицу, потому что:

- у одного товара может быть несколько складов;
- остатки меняются чаще карточки товара;
- складская истина находится в 1С;
- нужно показывать доступный остаток и дату синхронизации.

Критичная уникальность:

- `productId, warehouseId`.

### ProductPrice

Цена товара.

Ключевые поля:

- `productId`;
- `priceType`;
- `amount`;
- `currency`;
- `vatIncluded`;
- `vatRate`;
- `validFrom`;
- `validTo`;
- `syncBatchId`;
- `syncedAt`.

Цена вынесена отдельно от товара, потому что:

- цены обновляются независимо от карточки товара;
- могут появиться разные типы цен;
- цена должна фиксироваться снимком в заявке;
- сайт не является главным источником цены.

## 3. 1С и синхронизация

### OneCProductMapping

Таблица связи товара сайта с товаром 1С.

Ключевые поля:

- `productId`;
- `onecId`;
- `onecCode`;
- `onecName`;
- `rawData`;
- `syncedAt`.

Даже если `Product.onecId` заполнен, отдельная таблица связи оставляет место для будущих сценариев: несколько внешних кодов, смена кода 1С, миграции и аудит сопоставления.

### SyncBatch

Пакет синхронизации или импорта.

Ключевые поля:

- `source` - 1С, Excel, CSV, XML или ручной источник;
- `entity` - товар, цена, остаток, склад, статус заказа или документ;
- `status`;
- `startedAt`;
- `finishedAt`;
- счетчики строк: всего, успешно, пропущено, ошибок;
- `fileName`, `fileUrl`, `externalId`, `comment`.

Используется для импорта тестовых данных и регулярного обмена с 1С.

### SyncLog

Лог строк и событий синхронизации.

Ключевые поля:

- `syncBatchId`;
- `level`;
- `entity`;
- `entityId`;
- `rowNumber`;
- `message`;
- `details`.

В `details` нельзя хранить секреты, токены и пароли.

### IntegrationTask

Очередь интеграционных задач.

Ключевые поля:

- `system` - внешняя система;
- `type` - тип задачи;
- `status` - `PENDING`, `SENDING`, `SENT`, `RETRY_WAIT`, `FAILED`, `CANCELLED`;
- `entityType`;
- `entityId`;
- `requestId`;
- `idempotencyKey`;
- `attemptCount`;
- `maxAttempts`;
- `nextAttemptAt`;
- `lastAttemptAt`;
- `lastError`;
- `payload`;
- `response`.

Для отправки заявки в 1С задача должна иметь идемпотентный ключ, чтобы повторная отправка не создала дубль заказа.

### IntegrationLog

Технический журнал обмена с внешними системами.

Ключевые поля:

- `integrationTaskId`;
- `system`;
- `direction`;
- `entityType`;
- `entityId`;
- `status`;
- `attemptNumber`;
- `operation`;
- `endpoint`;
- `durationMs`;
- `externalId`;
- `errorCode`;
- `errorMessage`;
- `errorDetails`;
- `payloadHash`;
- `safePayload`;
- `safeResponse`.

В логах нельзя хранить секреты, токены, пароли и полные чувствительные данные. Для отладки хранится хэш полезной нагрузки и безопасный снимок без секретов.

## 4. Заявки

### Customer

Покупатель или контакт.

Ключевые поля:

- `userId`;
- `name`;
- `phone`;
- `email`.

Покупатель может быть связан с пользователем личного кабинета, но заявка может быть сохранена и без регистрации.

### CustomerCompany

Юридическое лицо покупателя.

Ключевые поля:

- `customerId`;
- `companyName`;
- `shortName`;
- `inn`;
- `kpp`;
- `ogrn`;
- адреса;
- контактные данные;
- банковские реквизиты.

Банковские реквизиты можно заполнять позже, если в первом рабочем релизе они не обязательны.

### Request

Заявка сайта.

Ключевые поля:

- `requestNumber`;
- `customerId`;
- `customerCompanyId`;
- `status`;
- `onecSendStatus`;
- `onecOrderId`;
- `onecOrderNumber`;
- `onecSentAt`;
- `onecUpdatedAt`;
- контактные данные;
- комментарии;
- итоговая сумма и валюта.

Заявка всегда сначала сохраняется в базе сайта. Только после этого создается `IntegrationTask` на отправку в 1С.

### RequestItem

Строка заявки.

Ключевые поля:

- `requestId`;
- `productId`;
- `lineNumber`;
- `requestedArticle`;
- `matchedArticle`;
- `normalizedArticle`;
- `productNameSnapshot`;
- `productConditionSnapshot`;
- `warehouseIdSnapshot`;
- `warehouseNameSnapshot`;
- `stockAvailableSnapshot`;
- `priceSnapshot`;
- `currencySnapshot`;
- `quantity`;
- `onecProductIdSnapshot`.

Таблица хранит снимок названия, цены, остатка, склада и состояния товара на момент отправки. Это нужно, потому что после заявки данные в 1С могут измениться.

### DeliveryInfo

Доставка по заявке.

Ключевые поля:

- `requestId`;
- `method`;
- `carrier`;
- `city`;
- `address`;
- `pickupPointId`;
- `terminalId`;
- получатель;
- вес и габариты;
- расчетная цена;
- комментарий.

Данные доставки передаются в 1С вместе с заявкой, если это предусмотрено интеграцией.

Ключевые поля:

Ключевые поля:

- `requestId`;
- `type`;
- `status`;
- `onecId`;
- `onecNumber`;
- `title`;
- `documentDate`;
- `validUntil`;
- `amount`;
- `currency`;
- `url`;
- `fileKey`;

Документы связаны с заявкой или заказом. У документа есть внешний идентификатор 1С. Сайт не формирует официальные документы самостоятельно.

## 7. Подготовка к маркетплейсу

### Seller

Минимальная сущность продавца.

Ключевые поля:

- `name`;
- `status`.

В первом рабочем релизе полноценный маркетплейс не реализуется. `Seller` нужен только как осторожный задел, чтобы в будущем можно было связать товар с продавцом без полной переделки каталога.

## 8. Какие данные приходят из 1С

Из 1С приходят:

- товары;
- артикулы и дополнительные идентификаторы;
- бренды;
- категории;
- состояние товара;
- изображения или ссылки на изображения;
- характеристики;
- склады;
- остатки;
- цены;
- статусы заказов;
- документы или ссылки на документы.

Эти данные на сайте являются локальной копией. Основное редактирование должно происходить в 1С.

## 9. Какие данные создаются на сайте

На сайте создаются:

- пользователи;
- покупатели;
- юридические лица покупателей;
- заявки;
- строки заявок;
- доставка заявки;
- интеграционные задачи;
- логи интеграций;
- тестовые импорты Excel/CSV до подключения 1С.

## 10. Что нельзя редактировать на сайте как источник истины

После подключения 1С нельзя считать сайт главным местом редактирования:

- номенклатуры;
- артикулов;
- цен;
- остатков;
- складов;
- официальных заказов;
- счетов;
- договоров;
- бухгалтерских документов.

Админка может показывать эти данные и помогать диагностировать ошибки, но не должна заменять 1С.

## 11. Поиск по артикулу

Поиск по артикулу работает через `ProductIdentifier`.

Алгоритм:

1. Пользователь вводит артикул.
2. Сайт нормализует ввод: убирает лишние пробелы, приводит к единому регистру, обрабатывает дефисы и разделители.
3. Сайт ищет `ProductIdentifier.normalizedValue`.
4. Точные совпадения получают приоритет.
5. Сайт загружает товар, состояние, цену, остаток и склад.
6. Если найдено несколько товаров, варианты показываются явно.

Похожее название товара не является основанием для автоматического аналога.

## 12. Массовый поиск

Массовый поиск использует тот же механизм, что и точный поиск.

Алгоритм:

1. Пользователь отправляет список артикулов.
2. Каждая строка нормализуется отдельно.
3. По каждой строке выполняется поиск в `ProductIdentifier`.
4. Результат возвращается построчно.
5. Для найденных товаров показываются цена, остаток, склад и состояние.
6. Ненайденные строки не отбрасываются, а возвращаются со статусом "не найдено".

Массовый поиск не должен молча объединять товары по похожим названиям.

## 13. Сохранение заявки

При отправке заявки сайт должен:

1. Создать или найти `Customer`.
2. Сохранить `CustomerCompany`, если покупатель указал юридическое лицо.
3. Создать `Request`.
4. Создать `RequestItem` по каждой позиции.
5. Сохранить снимок цены, остатка, склада, названия и состояния.
6. Сохранить `DeliveryInfo`, если доставка указана.
7. Создать `IntegrationTask` на отправку в 1С.

Даже если 1С недоступна, шаги сохранения заявки должны завершиться успешно.

## 14. Отправка заявки в 1С

Отправка заявки выполняется через `IntegrationTask`.

Порядок:

1. Задача создается в статусе `PENDING`.
2. Фоновый обработчик переводит ее в `SENDING`.
3. Сайт отправляет заявку в 1С с `idempotencyKey`.
4. При успехе задача получает `SENT`, а заявка получает `SENT_TO_ONE_C` или другой согласованный статус.
5. Сайт сохраняет `onecOrderId` и `onecOrderNumber`.
6. При ошибке задача получает `RETRY_WAIT` или `FAILED`, увеличивает `attemptCount`, заполняет `lastError` и `nextAttemptAt`.

## 15. Недоступность 1С

Если 1С недоступна:

- каталог продолжает работать по последней синхронизированной копии;
- поиск продолжает работать по локальной базе;
- заявка сохраняется в базе сайта;
- интеграционная задача остается в очереди;
- ошибка записывается в `IntegrationLog`;
- администратор видит проблему;
- повторная отправка выполняется позже.

Недоступность 1С не должна приводить к потере заявки.
