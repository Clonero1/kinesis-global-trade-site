# Дорожная карта разработки

## Этап 1. Проектирование базы данных и интеграционного контура

**Цель этапа:** зафиксировать структуру данных и границы ответственности сайта, 1С и внешних сервисов.

**Что должно быть сделано:**

- описана схема PostgreSQL;
- подготовлена Prisma-модель;
- определены сущности товаров, артикулов, цен, остатков, складов, заявок, пользователей и логов;
- описаны статусы интеграционных задач;
- зафиксированы правила идемпотентности при отправке заявок в 1С.

**Вероятно затрагиваемые файлы/модули:**

- `prisma/schema.prisma`;
- `src/modules/catalog`;
- `src/modules/requests`;
- `src/modules/integrations`;
- `src/modules/admin`;
- `src/jobs`.

**Критерии готовности:**

- схема покрывает каталог, заявки, юридические лица, доставку и интеграционные логи;
- артикул выделен как главный идентификатор поиска;
- состояние товара хранится отдельно;
- склад и остаток можно хранить по каждому товару;
- заявки можно сохранять до отправки в 1С.

**Риски:**

- неполные данные от 1С;
- неясные статусы заказов;
- неверное смешение ответственности сайта и 1С.

**Что нельзя делать на этом этапе:**

- писать публичный интерфейс вместо проектирования данных;
- делать сайт главным источником остатков;
- закладывать автоматическое объединение аналогов по названию.

## Этап 2. Базовый каталог

**Цель этапа:** сделать отображение товаров, категорий, карточки товара, цены, остатков и склада.

**Что должно быть сделано:**

- список товаров;
- карточка товара;
- отображение артикула;
- отображение состояния;
- отображение склада;
- отображение цены и остатка;
- базовая фильтрация по категории, бренду и состоянию.

**Вероятно затрагиваемые файлы/модули:**

- `src/app`;
- `src/app/catalog`;
- `src/app/product`;
- `src/modules/catalog`;
- `src/components/catalog`;
- `src/components/product`.

**Критерии готовности:**

- пользователь видит каталог;
- карточка товара показывает артикул, состояние, цену, остаток и склад;
- данные берутся из базы сайта;
- интерфейс не подразумевает редактирование данных как истины.

**Риски:**

- устаревшие остатки;
- нехватка изображений;
- неоднозначные категории.

**Что нельзя делать на этом этапе:**

- строить полноценный маркетплейс;
- добавлять кабинет продавца;
- скрывать склад в карточке товара.

## Этап 3. Поиск по артикулу

**Цель этапа:** реализовать точный поиск по артикулу как ключевой сценарий.

**Что должно быть сделано:**

- нормализация артикула;
- поиск по оригинальному и нормализованному артикулу;
- вывод точного совпадения;
- отдельный вывод близких совпадений, если они нужны;
- защита от ложного определения аналогов.

**Вероятно затрагиваемые файлы/модули:**

- `src/app/search`;
- `src/modules/search`;
- `src/modules/catalog`;
- `src/lib/article-normalization`;
- `prisma/schema.prisma`.

**Критерии готовности:**

- точный артикул находится быстро;
- результат показывает товар, состояние, цену, остаток и склад;
- похожие названия не считаются аналогами автоматически;
- поиск работает без обращения к 1С в момент запроса.

**Риски:**

- разные форматы артикулов;
- пробелы, дефисы и регистр;
- дубли артикулов с разным состоянием.

**Что нельзя делать на этом этапе:**

- заменять точный поиск полнотекстовым поиском по названию;
- автоматически объединять товары по похожему названию;
- скрывать случаи, когда найдено несколько вариантов.

## Этап 4. Массовый поиск товаров

**Цель этапа:** дать клиенту возможность вставить список артикулов и получить результат по каждой строке.

**Что должно быть сделано:**

- форма массового ввода;
- разбор строк;
- нормализация каждого артикула;
- поиск по локальной базе;
- отображение найденных и ненайденных позиций;
- добавление найденных позиций в заявку.

**Вероятно затрагиваемые файлы/модули:**

- `src/app/bulk-search`;
- `src/modules/bulk-search`;
- `src/modules/search`;
- `src/components/bulk-search`;
- `src/modules/requests`.

**Критерии готовности:**

- каждая входная строка получает отдельный результат;
- найденные товары можно добавить в заявку;
- ненайденные позиции сохраняются в результате;
- состояние и склад видны в результатах.

**Риски:**

- большие списки артикулов;
- дубли во входном списке;
- несколько товаров на один артикул.

**Что нельзя делать на этом этапе:**

- отбрасывать ненайденные строки;
- молча выбирать один товар из нескольких вариантов;
- считать товары аналогами по похожему названию.

## Этап 5. Импорт тестовых данных

**Цель этапа:** загрузить тестовые Excel/CSV-данные для разработки каталога до полноценной интеграции с 1С.

**Что должно быть сделано:**

- загрузка Excel/CSV;
- проверка колонок;
- импорт товаров, цен, остатков и складов;
- журнал импортов;
- отображение ошибок строк.

**Вероятно затрагиваемые файлы/модули:**

- `src/app/admin/imports`;
- `src/modules/import`;
- `src/modules/catalog`;
- `src/components/admin`;
- `prisma/schema.prisma`.

**Критерии готовности:**

- файл можно загрузить через админку;
- ошибки импорта видны администратору;
- данные появляются в каталоге;
- импорт помечен как тестовый источник.

**Риски:**

- формат файла не совпадает с будущим форматом 1С;
- тестовые данные начинают восприниматься как основной источник;
- грязные артикулы.

**Что нельзя делать на этом этапе:**

- заменять интеграцию с 1С тестовым импортом навсегда;
- делать Excel главным источником товаров;
- импортировать аналоги без явного признака.

## Этап 6. Прием заявок

**Цель этапа:** реализовать корзину или заявку и сохранять ее в базе сайта.

**Что должно быть сделано:**

- добавление товара в заявку;
- изменение количества;
- ввод контактов;
- ввод юридического лица;
- ввод доставки;
- сохранение заявки в PostgreSQL;
- создание интеграционной задачи на отправку.

**Вероятно затрагиваемые файлы/модули:**

- `src/app/cart`;
- `src/app/request`;
- `src/modules/requests`;
- `src/modules/customers`;
- `src/modules/integrations`;
- `src/app/admin/requests`.

**Критерии готовности:**

- заявка сохраняется даже без 1С;
- состав заявки фиксируется;
- цена и остаток сохраняются снимком на момент заявки;
- администратор видит заявку.

**Риски:**

- потеря заявки при ошибке;
- неполные контактные данные;
- неверное ожидание клиента по остаткам.

**Что нельзя делать на этом этапе:**

- отправлять заявку только напрямую в 1С без локального сохранения;
- формировать официальный счет на сайте;
- подключать онлайн-оплату.

## Этап 7. Интеграция с 1С

**Цель этапа:** подключить обмен товарами, ценами, остатками и заявками с 1С.

**Что должно быть сделано:**

- получение товаров из 1С;
- получение цен;
- получение остатков и складов;
- отправка заявок;
- очередь повторной отправки;
- логирование ошибок обмена;
- ручной повтор из админки.

**Вероятно затрагиваемые файлы/модули:**

- `src/integrations/onec`;
- `src/jobs/onec`;
- `src/modules/integrations`;
- `src/modules/catalog`;
- `src/modules/requests`;
- `src/app/admin/integrations`.

**Критерии готовности:**

- каталог обновляется из 1С;
- остатки и цены синхронизируются по расписанию;
- заявка передается в 1С;
- при недоступности 1С заявка остается в очереди;
- ошибки видны в админке.

**Риски:**

- неподготовленный API 1С;
- дубли заказов при повторной отправке;
- медленный обмен большими пакетами.

**Что нельзя делать на этом этапе:**

- делать сайт источником цен и остатков;
- создавать дубли заказов без идемпотентности;
- скрывать ошибки обмена.

## Этап 8. Интеграция с Битрикс24

**Цель этапа:** передавать CRM-события без нарушения роли 1С.

**Что должно быть сделано:**

- согласовать сценарий CRM;
- передавать лид или сделку;
- передавать контактные данные;
- передавать ссылку на заявку;
- логировать ошибки CRM-обмена.

**Вероятно затрагиваемые файлы/модули:**

- `src/integrations/bitrix24`;
- `src/modules/integrations`;
- `src/modules/requests`;
- `src/app/admin/integrations`.

**Критерии готовности:**

- заявка отражается в Битрикс24 по согласованному сценарию;
- ошибки видны администратору;
- данные 1С и Битрикс24 не конфликтуют.

**Риски:**

- дубли лидов;
- разные статусы в CRM и 1С;
- лишняя нагрузка на менеджеров.

**Что нельзя делать на этом этапе:**

- делать Битрикс24 источником товаров или остатков;
- заменять заказ 1С сделкой в CRM;
- отправлять непроверенные персональные данные без согласованной политики.

## Этап 9. Личный кабинет покупателя

**Цель этапа:** дать покупателю доступ к истории заявок, статусам и документам.

**Что должно быть сделано:**

- регистрация или вход;
- профиль покупателя;
- юридические лица покупателя;
- история заявок;
- статусы заказов из 1С;
- документы из 1С.

**Вероятно затрагиваемые файлы/модули:**

- `src/app/account`;
- `src/modules/auth`;
- `src/modules/customers`;
- `src/modules/requests`;
- `src/modules/documents`.

**Критерии готовности:**

- покупатель видит свои заявки;
- статусы соответствуют данным из 1С;
- документы доступны только владельцу заявки;
- юридические лица можно использовать в новых заявках.

**Риски:**

- доступ к чужим документам;
- неактуальные статусы;
- сложная авторизация.

**Что нельзя делать на этом этапе:**

- показывать документы без проверки прав;
- формировать документы на сайте;
- давать покупателю редактировать данные 1С напрямую.

## Этап 10. Документы из 1С

**Цель этапа:** отображать счета, договоры и другие документы, сформированные в 1С.

**Что должно быть сделано:**

- получение списка документов по заказу;
- хранение ссылок или файлов;
- отображение документов в кабинете;
- отображение документов в админке;
- обновление статусов документов.

**Вероятно затрагиваемые файлы/модули:**

- `src/modules/documents`;
- `src/integrations/onec`;
- `src/app/account/documents`;
- `src/app/admin/documents`;
- `src/storage`.

**Критерии готовности:**

- документ создается в 1С;
- сайт получает ссылку или файл;
- клиент может открыть документ;
- сайт не создает официальный документ самостоятельно.

**Риски:**

- ссылки из 1С недоступны клиенту;
- документы устаревают;
- ошибки прав доступа.

**Что нельзя делать на этом этапе:**

- генерировать счета и договоры на сайте;
- хранить документы без контроля доступа;
- изменять юридическое содержание документа.

## Этап 11. Доставка СДЭК и Деловые Линии

**Цель этапа:** добавить расчет и передачу доставки по заявке.

**Что должно быть сделано:**

- расчет доставки СДЭК;
- расчет доставки Деловых Линий;
- выбор способа доставки;
- передача доставки в 1С;
- получение трек-номера или статуса, если доступно.

**Вероятно затрагиваемые файлы/модули:**

- `src/integrations/cdek`;
- `src/integrations/dellin`;
- `src/modules/delivery`;
- `src/modules/requests`;
- `src/components/delivery`.

**Критерии готовности:**

- клиент видит варианты доставки;
- выбранный вариант сохраняется в заявке;
- 1С получает данные доставки;
- ошибки расчета не блокируют сохранение заявки.

**Риски:**

- неточные габариты товаров;
- изменение тарифов;
- ошибки внешних API.

**Что нельзя делать на этом этапе:**

- блокировать заявку, если расчет доставки недоступен;
- считать расчет окончательной бухгалтерской суммой без подтверждения 1С;
- смешивать доставку с товарными остатками.

## Этап 12. XML-выгрузка для Авито

**Цель этапа:** формировать XML-выгрузку товаров для Авито на основе данных из 1С.

**Что должно быть сделано:**

- выбрать товары для выгрузки;
- сформировать XML по требованиям Авито;
- добавить цены, остатки, изображения и описания;
- хранить историю генерации;
- логировать ошибки.

**Вероятно затрагиваемые файлы/модули:**

- `src/modules/avito`;
- `src/app/admin/avito`;
- `src/modules/catalog`;
- `src/storage`;
- `src/jobs/avito`.

**Критерии готовности:**

- XML генерируется по расписанию или вручную;
- данные берутся из локальной копии, полученной из 1С;
- ошибки видны администратору;
- файл доступен по защищенной или согласованной ссылке.

**Риски:**

- несоответствие требованиям Авито;
- устаревшие цены;
- отсутствие изображений.

**Что нельзя делать на этом этапе:**

- создавать отдельную независимую базу товаров для Авито;
- выгружать товары без контроля активности и остатков;
- менять данные 1С через Авито-выгрузку.

## Этап 13. Подготовка к маркетплейсу

**Цель этапа:** заложить архитектурную основу для будущего маркетплейса без реализации полного сценария.

**Что должно быть сделано:**

- описать модель продавца;
- определить связь продавца с товаром;
- описать будущие комиссии;
- описать модерацию;
- определить будущий кабинет продавца;
- проверить, что текущая архитектура не мешает расширению.

**Вероятно затрагиваемые файлы/модули:**

- `docs`;
- `prisma/schema.prisma`;
- `src/modules/marketplace`;
- `src/modules/catalog`;
- `src/modules/requests`.

**Критерии готовности:**

- есть проектное описание будущего маркетплейса;
- текущая архитектура не переписана под продавцов преждевременно;
- 1С остается главным источником на текущем этапе.

**Риски:**

- преждевременное усложнение;
- смешение товаров 1С и товаров продавцов;
- изменение бизнес-модели до проверки базового сайта.

**Что нельзя делать на этом этапе:**

- реализовывать полноценный маркетплейс;
- делать кабинет продавца;
- добавлять сложные комиссии;
- переносить товарную истину с 1С на сайт.
