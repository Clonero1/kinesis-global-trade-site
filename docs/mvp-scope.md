# Первый рабочий релиз

## 1. Цель

Первый рабочий релиз должен дать компании работающий сайт-витрину с каталогом, поиском по артикулу, массовым поиском, приемом заявок и базовым административным контуром.

Главная цель релиза - проверить ключевые пользовательские сценарии и интеграционную архитектуру без преждевременного перехода к полноценному маркетплейсу.

## 2. Обязательно включить

В первый рабочий релиз обязательно входят:

- главная страница;
- каталог;
- карточка товара;
- точный поиск по артикулу;
- массовый поиск товаров списком;
- загрузка тестового Excel/CSV;
- заявка или корзина;
- сохранение заявки в базе сайта;
- передача заявки в 1С через интеграционный слой или временную заглушку;
- лог ошибок интеграции;
- базовая админка товаров, импортов и заявок.

## 3. Не включать

В первый рабочий релиз не входят:

- полноценный маркетплейс;
- кабинет продавца;
- онлайн-оплата;
- сложная система комиссий;
- автоматическое формирование документов на сайте;
- автоматическое объединение аналогов по названию.

## 4. Ключевые правила релиза

Артикул является главным идентификатором товара. Поиск по артикулу должен быть точным, быстрым и предсказуемым.

Товары нельзя автоматически считать аналогами только по похожему названию. Если точного совпадения нет, сайт может показать близкие варианты только как отдельную подсказку без утверждения, что это аналоги.

Состояние товара должно храниться отдельно от названия и описания. Минимальный набор состояний:

- новый;
- б/у;
- восстановленный;
- под заказ.

Склад должен отображаться:

- в карточке товара;
- в результатах поиска;
- в результатах массового поиска;
- в админке.

Сайт должен сохранять заявку даже при недоступности 1С. После сохранения заявка должна попасть в очередь отправки или во временный интеграционный контур.

1С является главным источником товаров, цен, остатков, складов и документов. Сайт хранит рабочую копию данных для отображения и пользовательских сценариев.

## 5. Состав пользовательских сценариев

Покупатель должен иметь возможность:

- открыть главную страницу;
- перейти в каталог;
- посмотреть карточку товара;
- найти товар по точному артикулу;
- вставить список артикулов для массового поиска;
- увидеть найденные и ненайденные позиции;
- добавить найденные товары в заявку;
- заполнить контактные данные;
- заполнить данные юридического лица, если нужно;
- указать доставку или комментарий по доставке;
- отправить заявку;
- получить подтверждение, что заявка сохранена.

Администратор должен иметь возможность:

- загрузить тестовый Excel/CSV;
- увидеть результат импорта;
- посмотреть список товаров;
- посмотреть заявки;
- увидеть статус отправки заявки в 1С или временную заглушку;
- увидеть ошибки интеграции;
- повторить отправку заявки вручную, если это реализовано в рамках базовой админки.

## 6. Данные первого рабочего релиза

Для товара нужно хранить:

- артикул;
- нормализованный артикул;
- название;
- бренд;
- категорию;
- состояние;
- описание;
- цену;
- остаток;
- склад;
- признак активности.

Для заявки нужно хранить:

- номер заявки сайта;
- дату создания;
- контактные данные;
- юридическое лицо, если указано;
- состав товаров;
- количество;
- снимок цены;
- снимок остатка;
- склад;
- доставку;
- комментарий;
- статус отправки в 1С или временную заглушку.

Для интеграционного лога нужно хранить:

- дату операции;
- направление обмена;
- тип сущности;
- идентификатор заявки или товара;
- статус;
- текст ошибки;
- номер попытки;
- дату следующей попытки, если есть.

## 7. Критерии готовности

Первый рабочий релиз считается готовым, если:

- каталог открывается и показывает тестовые товары;
- карточка товара показывает артикул, состояние, цену, остаток и склад;
- точный поиск по артикулу возвращает ожидаемый товар;
- массовый поиск возвращает результат по каждой строке;
- тестовый Excel/CSV загружается через админку;
- заявка сохраняется в базе сайта;
- недоступность 1С или заглушки не приводит к потере заявки;
- ошибка интеграции попадает в лог;
- администратор может увидеть товары, импорты и заявки;
- сайт не формирует официальные документы самостоятельно.

## 8. Границы релиза

Первый рабочий релиз должен быть достаточно простым, чтобы быстро проверить рабочую ценность сайта, но достаточно надежным, чтобы не терять заявки.

Главные ограничения:

- документы формируются только в 1С;
- товарная истина находится в 1С;
- складская истина находится в 1С;
- бухгалтерская истина находится в 1С;
- сайт не редактирует основную номенклатуру;
- сайт не объединяет товары в аналоги автоматически.
