# Техническое задание на интеграцию сайта с 1С

## 1. Назначение интеграции

Интеграция с 1С нужна для обмена между сайтом и основной учетной системой компании.

1С является главным источником:

- товаров;
- номенклатуры;
- артикулов;
- цен;
- остатков;
- складов;
- заказов;
- счетов;
- договоров;
- официальных документов;
- статусов заказов.

Сайт является витриной, каталогом, инструментом поиска, точкой приема заявок и интеграционным слоем. Сайт не должен становиться главным источником складской или бухгалтерской истины.

## 2. Направления обмена

### 2.1. Данные, которые сайт получает из 1С

Сайт должен получать из 1С:

- товары и номенклатуру;
- артикулы и дополнительные идентификаторы;
- категории или товарные группы;
- бренды и производителей;
- состояние товара;
- описания и характеристики;
- изображения или ссылки на изображения;
- склады;
- остатки по складам;
- цены;
- валюты;
- ставки НДС, если они нужны для отображения или заявки;
- признак активности товара;
- статусы заказов;
- номера заказов в 1С;
- счета;
- договоры;
- ссылки на документы;
- файлы документов, если 1С отдает их напрямую;
- трек-номера и статусы доставки, если они ведутся в 1С.

### 2.2. Данные, которые сайт передает в 1С

Сайт должен передавать в 1С:

- заявку клиента;
- уникальный идентификатор заявки сайта;
- дату и время создания заявки;
- источник заявки;
- состав товаров;
- количество по каждой позиции;
- снимок цены на момент заявки;
- контактные данные покупателя;
- данные юридического лица;
- выбранный способ доставки;
- адрес, терминал или пункт выдачи;
- комментарий клиента;
- комментарий менеджера, если он добавлен в админке;
- технические признаки для идемпотентности и повторной отправки.

## 3. Частота обновления данных

Рекомендуемая частота обмена:

| Данные | Частота | Комментарий |
| --- | --- | --- |
| Товары и номенклатура | Полная синхронизация 1 раз в сутки, инкрементальная каждые 1-2 часа | Частоту нужно подтвердить со специалистом по 1С |
| Цены | Каждые 15-60 минут | Если цены меняются редко, интервал можно увеличить |
| Остатки | Каждые 5-15 минут | Остатки наиболее чувствительны к устареванию |
| Склады | 1 раз в сутки и при изменениях | Склады меняются редко |
| Статусы заказов | Каждые 5-15 минут или по webhook из 1С | Предпочтителен push-сценарий, если 1С его поддерживает |
| Документы | По событию создания документа или каждые 5-15 минут | Сайт должен получать номер, статус, ссылку или файл |

На первом этапе можно использовать регламентный обмен по расписанию. Если 1С поддерживает события или webhook, для статусов и документов лучше использовать событийную отправку.

## 4. Обработка недоступности 1С

Если 1С недоступна, сайт должен продолжать работать в ограниченном режиме:

- каталог отображается по последней успешно полученной копии данных;
- поиск по артикулу работает по локальной базе PostgreSQL;
- массовый поиск работает по локальной базе PostgreSQL;
- карточка товара показывает последние известные цены и остатки;
- заявка клиента сохраняется в базе сайта;
- отправка заявки в 1С ставится в очередь;
- администратор видит статус обмена и ошибку;
- после восстановления 1С очередь автоматически повторяет отправку.

Сайт не должен удалять или терять заявку из-за ошибки обмена с 1С.

Для администраторов нужно показывать:

- дату последней успешной синхронизации товаров;
- дату последней успешной синхронизации цен;
- дату последней успешной синхронизации остатков;
- количество заявок, ожидающих отправки;
- количество ошибок обмена.

## 5. Очередь повторной отправки заявок

Каждая заявка после сохранения в базе сайта должна получить интеграционную задачу на отправку в 1С.

Рекомендуемые статусы задачи:

- pending;
- sending;
- sent;
- retry_wait;
- failed;
- cancelled.

Рекомендуемый порядок работы:

1. Сайт сохраняет заявку в PostgreSQL.
2. Сайт создает задачу отправки в 1С.
3. Фоновый обработчик берет задачу из очереди.
4. Перед отправкой обработчик присваивает заявке идемпотентный ключ, например `site_request_id`.
5. 1С должна использовать этот ключ, чтобы не создавать дубли при повторной отправке.
6. При успешной отправке сайт сохраняет номер или идентификатор заказа 1С.
7. При ошибке сайт сохраняет текст ошибки и планирует повторную попытку.

Рекомендуемая схема повторов:

| Попытка | Интервал |
| --- | --- |
| 1 | Сразу |
| 2 | Через 1 минуту |
| 3 | Через 5 минут |
| 4 | Через 15 минут |
| 5 | Через 60 минут |
| Далее | 1 раз в час до ручного вмешательства или лимита |

Максимальное количество попыток должно быть настраиваемым.

Администратор должен иметь возможность:

- увидеть ошибку;
- повторить отправку вручную;
- отменить отправку;
- отметить заявку как обработанную вручную, если она уже создана в 1С.

## 6. Логирование ошибок обмена

Все операции обмена с 1С должны логироваться.

Минимальные поля лога:

- идентификатор записи лога;
- дата и время;
- направление обмена;
- тип сущности;
- идентификатор сущности на сайте;
- идентификатор сущности в 1С, если известен;
- статус операции;
- номер попытки;
- код ошибки;
- текст ошибки;
- техническое описание ошибки;
- длительность запроса;
- адрес метода или имя операции;
- хэш или безопасный снимок полезной нагрузки;
- пользователь или системный процесс, инициировавший операцию.

В логах нельзя хранить секреты, токены доступа, пароли и полные банковские реквизиты без необходимости.

## 7. Получение статусов заказов

После успешной отправки заявки в 1С сайт должен получить внешний идентификатор заказа или заявки в 1С.

Дальше статусы могут обновляться одним из способов:

- сайт периодически запрашивает статусы у 1С;
- 1С отправляет изменение статуса на endpoint сайта;
- комбинированный вариант: webhook плюс периодическая сверка.

Минимальные данные статуса:

- идентификатор заказа сайта;
- идентификатор заказа 1С;
- номер заказа 1С;
- статус;
- дата изменения статуса;
- комментарий менеджера или системы;
- признак необходимости действия со стороны клиента.

Рекомендуемые статусы для сайта:

- заявка создана;
- отправляется в 1С;
- передана в 1С;
- принята в работу;
- ожидает оплаты;
- оплачена;
- собирается;
- готова к отгрузке;
- отгружена;
- завершена;
- отменена;
- ошибка обмена.

Финальное соответствие статусов нужно согласовать с 1С.

## 8. Получение счетов, договоров и документов

Официальные документы формируются в 1С. Сайт не формирует их самостоятельно.

Сайт должен получать из 1С:

- тип документа;
- номер документа;
- дату документа;
- статус документа;
- ссылку на документ или файл;
- срок действия, если применимо;
- сумму, валюту и НДС, если применимо;
- идентификатор связанного заказа;
- дату последнего обновления.

Если 1С отдает ссылку, сайт хранит ссылку и показывает ее в личном кабинете.

Если 1С отдает файл, сайт может сохранить файл в S3-совместимом хранилище и хранить в базе метаданные файла.

Документы должны быть доступны:

- в админке заявки;
- в личном кабинете покупателя;
- в истории заказа.

## 15. Вопросы к специалисту по 1С перед разработкой

Перед разработкой нужно согласовать:

1. Какая конфигурация 1С используется.
2. Есть ли уже готовый HTTP/API-обмен.
3. Какие протоколы доступны: REST, SOAP, CommerceML, OData, файлы, другой вариант.
4. Кто дорабатывает 1С под обмен с сайтом.
5. Какой справочник является источником номенклатуры.
6. Как в 1С хранится основной артикул.
7. Есть ли дополнительные артикулы, кросс-номера и заменители.
8. Как в 1С хранится состояние товара: новый, б/у, восстановленный, под заказ.
9. Как в 1С хранится склад и остаток по складу.
10. Какие типы цен нужно передавать на сайт.
11. Нужна ли авторизация клиента для получения индивидуальных цен.
12. Как часто реально можно отдавать остатки.
13. Есть ли инкрементальная выгрузка измененных товаров.
14. Как 1С будет принимать заявку с сайта.
15. Как 1С должна защищаться от дублей заявок.
16. Какой номер или идентификатор 1С вернет сайту после создания заказа.
17. Какие статусы заказов нужно показывать клиенту.
18. Когда создаются счет, договор и другие документы.
19. Как 1С будет отдавать документы: ссылка, файл, печатная форма, внешний сервис.
20. Какие поля юридического лица обязательны для создания заказа.
21. Какие поля доставки обязательны для создания заказа.
22. Кто отвечает за качество товарных названий, артикулов и остатков.
23. Что делать с товарами, которые исчезли из выгрузки.
24. Нужно ли передавать на сайт изображения из 1С или из отдельного хранилища.
25. Какие ограничения есть по нагрузке, времени обмена и размеру пакетов.
